Data Modelling & SQL Basics

Defining Frameworks, Syntax Dialects, and Foundational Query Building Blocks

1. Defining SQL & DBMS

To architect enterprise data layers, we must establish a clear boundary between standard interface patterns and implementation technologies:

Structured Query Language (SQL)

A standardized declarative programming language designed specifically for managing, filtering, updating, and extracting structured relational tabular datasets.

Database Management System (DBMS)

The underlying software platform that governs data storage engine paths, enforces constraint rules, handles concurrency multi-threading, and executes optimization plans.

2. The Four SQL Command Types

SQL commands are segmented into distinct functional families that specify what operational scope a given script runs against:

Family Scope Purpose Core Keywords Included Data Analyst Application Context
DDL Data Definition Language CREATE, ALTER, DROP, TRUNCATE Used to build, alter, or remove database structural schemas, tables, and constraint bridges.
DML Data Manipulation Language INSERT, UPDATE, DELETE Used to control state changes by inserting fresh observations, updating data, or purging records.
DQL Data Query Language SELECT Used to extract target attributes and run multi-table aggregations. The core focus of analysts.
DCL Data Control Language GRANT, REVOKE Used to manage administrative permissions, access levels, and security configurations on objects.

3. Relational SQL Dialects

While ANSI SQL forms the shared core standard, different database platforms utilize unique functional accents, optimizations, and syntax characteristics:

PostgreSQL

Enterprise Open-Source

Renowned for strict constraint enforcement, JSON data compliance, and extensive window analytical syntax options.

MySQL

Web Scale Core

The backend layer powering the web. Offers optimized query speeds for write-heavy data landing frames.

T-SQL (MSSQL)

Transact-SQL / Corporate

Microsoft's extension dialect. Includes control flow variables, error logging blocks, and advanced window functions.

SQLite

Serverless Embedded

A serverless file-backed binary database. Used for local analytics pipelines (and this interactive course workbook).

Oracle SQL

PL-SQL / Legacy Heavy

The enterprise banking database framework. Uses strong data access validation architectures.

4. IDE Environments & Integrated Application Pipelines

When working out in production, analysts and engineering teams match target environments with highly specific query developer IDEs, integrated closely into programming code bases:

Target IDE Environment Optimal Implementation Project Case Cross-Language App Integration Context
DBeaver / DataGrip Multi-Platform Agnostic Data Analytics: Ideal when managing hybrid cloud models combining PostgreSQL, MySQL, and Oracle architectures in a singular environment. Used by analytical data teams to map ERD schemas out prior to embedding SQL scripts directly inside Python pandas pipelines or **dbt** (Data Build Tool) nodes.
VS Code (SQL Extensions) Full-Stack Scripting & Git Pipelines: Best when queries coexist seamlessly right inside backend engineering structures, or microservices. Enables simultaneous testing of SQL statements and native application layers—such as backend integrations utilizing Node.js (pg-pool) or Python (SQLAlchemy).
SSMS / pgAdmin Native Administrative Tuning: Specialized tools tied directly to platform specific database optimization engines (SQL Server / PostgreSQL). Crucial for viewing internal query execution maps, setting up server cron jobs, and inspecting structural tables for heavy **Enterprise Apps (.NET Framework)**.

5. Relational Normalization & Keys

In database design, tables represent entities. Connections between tables represent explicit structural constraints.

  • Primary Key (PK): A unique structural column that explicitly identifies each row inside a table. Can never accept NULL targets.
  • Foreign Key (FK): A tracking attribute in a table that references a primary key index located in an external table, enforcing relational consistency.
  • Normalization (1NF, 2NF, 3NF): Minimizing structural data duplication by splitting broad matrices into distinct functional entities:
    • 1NF: Single atomic cell values inside rows, zero repeating column vectors.
    • 2NF: 1NF + elimination of partial key dependencies (all metrics must depend on the complete Primary Key block).
    • 3NF: 2NF + elimination of transitive dependencies (no non-key data attributes can depend on other non-key columns).

6. Relational Join Frameworks & Map

Joins define how distinct relational tables intersect when building unified multi-table query views:

INNER JOIN

Returns only rows that match keys across both tables.

LEFT JOIN

Returns all Left records, filling missing right inputs with NULL.

RIGHT JOIN

Returns all Right records, filling missing left inputs with NULL.

FULL OUTER JOIN

Combines all tables, keeping rows regardless of unmatched key paths.

7. SQL Query Building Blocks

Relational databases are queried using structured clauses. Understanding their logical execution order is key to writing bug-free SQL:

Step Clause Description Trainee Reinforcement Tip
1 FROM & JOIN Identifies source tables and joins rows based on matching key linkages. Always verify your primary-to-foreign key relation in the ON block to prevent query multiplying.
2 WHERE Filters individual rows before any grouping or aggregates are computed. Use standard operators like =, LIKE, IN, and IS NULL. You cannot use aggregate functions (like SUM) here.
3 GROUP BY Collapses multiple rows into grouped records based on matching column values. Every non-aggregated column in your SELECT list must be included in the GROUP BY clause.
4 HAVING Filters grouped results after aggregation has taken place. Used for conditional filters on aggregates (e.g., HAVING COUNT(*) > 1).
5 SELECT Specifies exactly which columns or expressions to project in the final output. Aliasing defined here (using AS) is not accessible in WHERE or GROUP BY clauses since they run earlier!

Querying Data Using Relationships

Unlocking Subqueries, Common Table Expressions (CTEs), and Derived Tables

Session 2 Concepts

When query complexity grows, using simple joins becomes insufficient. SQL provides tools to structure nested datasets, optimize aggregations, and reuse temporary results.

Subqueries
A query nested inside another query. Can be Scalar (returns 1 value), Multi-row (returns a column list for IN), or Correlated (references the outer query).
Derived Tables
A subquery nested inside the FROM clause that acts as a temporary, query-scoped table. Every derived table must be assigned an alias in SQL.
CTEs (Common Table Expressions)
Defined using the WITH keyword. CTEs construct a temporary, named result set that increases query readability and can be referenced multiple times sequentially.
Materialized Views
A query cache physically stored on disk as a table for high performance. Must be refreshed when underlying data changes (can be emulated via CREATE TABLE AS).

Traditional Subqueries vs. CTEs Comparison

Here is how the same operational question—finding wells whose total actual cost exceeds the database average—is solved using both methods. Notice how the subqueries in the FROM clause on the left act as Derived Tables (aliased as WellSummaries and AverageSub):

Traditional Nested Subquery Approach

Logic is nested "inside-out". Harder to read and maintain due to redundant code groupings.

SELECT well_name, total_actual_cost
FROM (
    SELECT w.well_name, SUM(dpd.actual_cost) AS total_actual_cost
    FROM WELL w
    JOIN WELL_PHASE wp ON w.well_id = wp.well_id
    JOIN DRILLING_PHASE_DETAILS dpd ON wp.well_phase_id = dpd.well_phase_id
    GROUP BY w.well_name
) AS WellSummaries
WHERE total_actual_cost > (
    SELECT AVG(total_cost)
    FROM (
        SELECT SUM(dpd2.actual_cost) AS total_cost
        FROM WELL_PHASE wp2
        JOIN DRILLING_PHASE_DETAILS dpd2 ON wp2.well_phase_id = dpd2.well_phase_id
        GROUP BY wp2.well_id
    ) AS AverageSub
);
Common Table Expression (CTE) Approach

Sequential, modular logic defined using WITH. Highly readable, clean, and reusable.

WITH WellCosts AS (
    SELECT w.well_name, SUM(dpd.actual_cost) AS total_actual_cost
    FROM WELL w
    JOIN WELL_PHASE wp ON w.well_id = wp.well_id
    JOIN DRILLING_PHASE_DETAILS dpd ON wp.well_phase_id = dpd.well_phase_id
    GROUP BY w.well_name
),
AverageWellCost AS (
    SELECT AVG(total_actual_cost) AS avg_cost
    FROM WellCosts
)
SELECT wc.well_name, wc.total_actual_cost
FROM WellCosts wc
CROSS JOIN AverageWellCost awc
WHERE wc.total_actual_cost > awc.avg_cost;

For complex datasets, we can cache values physically in a table to bypass on-the-fly calculations. Note: SQLite does not support native Materialized Views. Here is the actual standard SQL syntax for reference, alongside the SQLite emulation:

JOIN WELL w ON wp.well_id = w.well_id
JOIN RIG r ON w.rig_id = r.rig_id
GROUP BY w.well_name, r.rig_name;

-- Query the emulated Materialized View

ORDER BY total_actual_cost DESC;

WITH CombinedCosts AS (
    SELECT well_phase_id, plan_cost, actual_cost FROM DRILLING_PHASE_DETAILS
    UNION ALL
    SELECT well_phase_id, plan_cost, actual_cost FROM COMPLETION_PHASE_DETAILS
)
SELECT p.phase_name, 
       ROUND(SUM(c.plan_cost), 2) as total_plan,
       ROUND(SUM(c.actual_cost), 2) as total_actual,
       ROUND(SUM(c.actual_cost - c.plan_cost), 2) as variance
FROM WELL_PHASE wp

JOIN CombinedCosts c ON wp.well_phase_id = c.well_phase_id
GROUP BY p.phase_name
ORDER BY variance DESC;

Aggregation & Analytical SQL

Mastering SQL Window Functions, Partitions, and Running Totals

Window Functions

Unlike regular GROUP BY functions which collapse multiple rows into one row, window functions perform calculations across a set of rows while keeping each individual row in the output.

Key Functions
ROW_NUMBER(): Assigns a unique sequential number to rows within a partition.
RANK(): Assigns rank with gaps if duplicates exist.
DENSE_RANK(): Assigns rank without gaps for duplicate values.
SUM() OVER: Running total calculation across a partition ordered by date.

       ), 2) as running_total_cost
FROM DRILLING_PHASE_DETAILS dpd
JOIN REPORT_JOURNAL rj ON dpd.report_journal_id = rj.report_journal_id
JOIN WELL_PHASE wp ON dpd.well_phase_id = wp.well_phase_id
JOIN WELL w ON wp.well_id = w.well_id
JOIN PHASE p ON wp.phase_id = p.phase_id
ORDER BY w.well_name, p.phase_name, rj.report_datetime;

Data Integrity & Validation

Auditing the Database for Duplicates, Orphan Records, and NULL Fields

Database Auditing & Views

Data imported from the field is often dirty. Relational database rules can sometimes be violated or bypassed during bulk loads, resulting in anomalies.

  • Duplicates: Identical entries logged multiple times (e.g. repeated daily reports).
  • Orphan Records: Child records pointing to parent IDs that no longer exist (violates FK rules). We use a LEFT JOIN to find these by returning all records on the left, and filtering for NULL on the right side.
  • Views: Stored SELECT statements that act as virtual tables, masking complex cleanup queries from downstream users.

Example 1: Identifying Duplicates (GROUP BY & HAVING) Toggle

Identifies duplicate records in REPORT_JOURNAL where multiple entries share the exact same well phase, datetime, and report type:

SELECT well_phase_id, report_datetime, report_type, COUNT(*) as occurrence_count
FROM REPORT_JOURNAL
GROUP BY well_phase_id, report_datetime, report_type
HAVING COUNT(*) > 1;
